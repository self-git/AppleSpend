# V2 第二版细化实现计划

## 1. 背景与当前基线

主计划中的 V2 目标是把 Apple 消费账本从“消费统计工具”升级为“Apple 生态资产与订阅管理工具”。当前 V1 已具备以下基础能力：

- CSV 导入与文件类型识别。
- App Store / Apple Store 两类核心 CSV 解析。
- IndexedDB 本地存储。
- Dashboard、Ledger、Assets 三个核心页面。
- 增量导入与重复跳过。
- Ledger 交易详情抽屉。
- 基础图片 resolver 与硬件图片精确匹配雏形。

V2 不应推倒重写，而应在现有 V1 数据模型、解析器、存储层和页面结构上继续扩展。

## 2. V2 总目标

V2 要回答这些问题：

1. 我的 Apple 生态长期订阅到底花了多少钱？
2. Store Credit 充值、消费和当前余额是否能完整追踪？
3. 我买过的设备、维修、AppleCare 和使用周期能否串起来？
4. Apple 数据导出的更多文件能否一次导入，而不是手动挑 CSV？
5. 是否能生成年度 Apple 消费报告？
6. 数据质量问题是否能被发现、解释和修正？

## 3. V2 范围

### 3.1 必做范围

- 支持 Apple 导出 ZIP / 多文件导入。
- 扩展文件识别器，基于表头识别更多 CSV。
- 新增订阅中心。
- 新增 Store Credit 中心。
- 扩展设备生命周期，关联 AppleCare、维修、设备信息。
- 新增年度报告页面。
- 新增数据质量与冲突处理页面。
- 新增规则配置页面。
- 支持完整数据备份和脱敏导出。

### 3.2 暂不做范围

- 不做云同步。
- 不做多用户账户。
- 不上传账单到服务端。
- 不做自动联网抓取 Apple 账户数据。
- 不承诺解析所有 Apple 导出文件，只覆盖 V2 目标需要的核心文件。
- 不在第一阶段引入 Tauri，除非 ZIP / 文件夹导入在浏览器端体验明显不足。

## 4. 推荐实施原则

### 4.1 数据层先行

V2 的页面都依赖新数据源。先把导入、解析、统一模型、存储和统计口径稳定下来，再做页面。

### 4.2 保持本地优先

所有解析、清洗、统计和报告生成默认在浏览器本地完成。导出 JSON、Markdown、PNG、PDF 也应优先走本地能力。

### 4.3 不破坏 V1

现有 Dashboard、Ledger、Assets 不能因 V2 数据源扩展而退化。新增模型应兼容旧数据，必要时通过 Dexie 版本迁移处理。

### 4.4 规则可解释

订阅、Store Credit、现金支出、账单项目价值、设备状态都需要能解释来源与口径。复杂规则不能只藏在代码里。

## 5. 数据源扩展计划

### 5.1 文件类型扩展

当前 `file-detector.ts` 只识别：

- `media_transactions`
- `online_purchase_history`

V2 建议扩展为：

```ts
export type AppleCsvFileType =
  | 'media_transactions'
  | 'online_purchase_history'
  | 'subscription_history'
  | 'billing_projection'
  | 'apple_pay_billings'
  | 'store_credit_history'
  | 'applecare_repairs'
  | 'applecare_cases'
  | 'apple_id_device_information'
  | 'unknown'
```

### 5.2 表头识别策略

不要只靠文件名。每种文件建立一组 required fields 和 optional fields：

```ts
interface AppleFileSignature {
  type: AppleCsvFileType
  label: string
  requiredFields: string[]
  optionalFields?: string[]
  pathHints?: string[]
}
```

识别顺序：

1. 先按 required fields 精确匹配。
2. 再按 pathHints / 文件名辅助确认。
3. 如果多个类型同时命中，标记为 ambiguous，进入数据质量提示。
4. 未识别文件不报错中断，只进入导入摘要。

### 5.3 ZIP / 多文件导入

建议新增依赖：

- `jszip`：浏览器端解析 ZIP。

导入入口保持一个组件，但内部处理三类输入：

- 单个 CSV。
- 多个 CSV。
- Apple 数据导出 ZIP。

建议新增模块：

```text
src/features/import/
  archive-reader.ts
  file-signatures.ts
  import-orchestrator.ts
```

职责：

- `archive-reader.ts`：从 ZIP 中提取 CSV 文件。
- `file-signatures.ts`：维护文件识别签名。
- `import-orchestrator.ts`：统一调度各种 parser，生成 ImportResult。

### 5.4 导入摘要升级

V2 导入摘要要按文件类型分组：

- 已识别并解析成功。
- 已识别但存在警告。
- 未识别。
- 跳过重复。
- 可能冲突。

导入完成后，用户应该知道“哪些数据进入了系统，哪些没进入，为什么”。

```ts
this.version(2).stores({

  assets: '&id, category, purchaseDate, orderNumber, status',
  importBatches: '&id, importedAt',
  subscriptions: '&id, serviceName, status, latestEventDate',
  subscriptionEvents: '&id, subscriptionId, date, eventType',
  storeCreditEntries: '&id, date, direction, relatedTransactionId',
  deviceLifecycles: '&assetId, status, serialNumber, isPrimaryDevice',
```

迁移原则：

- 不清空 V1 数据。
- 旧数据缺字段时给默认值。
- 新表从空表开始，由重新导入或 V2 parser 补齐。

## 7. Parser 实施细节

### 7.1 Subscription History parser

目标：

- 将订阅历史转成 `AppleSubscriptionEvent`。
- 聚合成 `AppleSubscription`。
- 与 App Store 交易记录按订单号、日期、服务名关联。

关键难点：

- 服务名可能在不同文件中写法不同。
- 免费试用、续订、取消、到期需要事件归一。
- 同一订阅可能多次取消再开通。

验收：

- 能看到每个订阅的时间线。
- 能计算累计支出和最近一次扣费。
- 能识别 active / expired / unknown。

### 7.2 Store Credit Transaction History parser

目标：

- 建立余额流水。
- 区分充值、消费、退款、调整。
- 与 App Store 交易中的 Store Credit 消费关联。

关键难点：

- Store Credit CSV 字段可能没有完整余额。
- 某些余额消费只在媒体服务账单中出现。
- 充值和消费不能重复计入现金支出。

验收：

- Store Credit 中心显示累计充值、已消费、估算余额。
- 同一笔 Store Credit 消费能跳转到 Ledger 交易。
- 无法关联的流水进入数据质量中心。

### 7.3 Billing / Apple Pay parser

目标：

- 用于交叉核对实际支付。
- 不默认生成新的交易，避免重复计入。
- 作为 `PaymentEvidence` 或 data quality source 参与核对。

建议模型：

```ts
export interface PaymentEvidence {
  id: string
  date: string
  amount: string
  currency: string
  paymentMethod?: string
  orderNumber?: string
  relatedTransactionIds: string[]
  raw: Record<string, string>
}
```

验收：

- 能识别可能金额不一致的订单。
- 不影响现有 Dashboard 总额，除非用户显式启用核对规则。

### 7.4 AppleCare / Repairs parser

目标：

- 解析维修、服务、支持案例。
- 尝试与资产关联。
- 关联失败进入数据质量中心。

关联策略：

1. 订单号 / 序列号精确匹配。
2. 设备名称和日期范围匹配。
3. 用户手动确认。

验收：

- Assets 设备详情中能看到维修 / 支持时间线。
- 无法关联的维修记录不会丢失。

### 7.5 Apple ID Device Information parser

目标：

- 补充设备出现过的记录。
- 识别设备名称、型号、系统、最后活动时间。
- 与 Apple Store 购买资产合并。

验收：

- 能看到“购买过但不再出现”和“出现过但没有购买记录”的设备差异。
- 这些差异进入数据质量页面，而不是静默忽略。

## 8. 页面与路由计划

### 8.1 路由新增

```text
/subscriptions
/store-credit
/reports
/data-quality
/settings/rules
```

导航建议：

- 主导航保留 Import、Dashboard、Ledger、Assets。
- V2 新页面可以放在二级导航或更多菜单中，避免顶部过载。

### 8.2 Subscriptions 页面

核心模块：

- 顶部指标：订阅累计支出、当前活跃数量、月均成本、年化成本。
- 订阅列表：服务名、状态、最近扣费、累计支出、周期。
- 时间线：开通、试用、续订、取消、到期。
- 分类筛选：Apple 服务 / 第三方 App / 未识别。

交互：

- 点击订阅进入详情抽屉。
- 详情抽屉显示原始事件和关联交易。
- 支持标记订阅状态为手动修正。

### 8.3 Store Credit 页面

核心模块：

- 累计充值。
- 已消费。
- 退款 / 调整。
- 当前估算余额。
- 余额流水时间线。
- 消费去向分类。

视觉要求：

- 参考当前 Dashboard 中 Apple Store 充值卡片，但去掉冗余解释文字。
- 口径说明放到折叠说明或信息提示，不占据主指标卡。

### 8.4 Assets 增强

新增能力：

- 设备详情抽屉。
- 购买记录、AppleCare、维修、Apple ID 出现记录合并展示。
- 手动维护出售价格、出售日期、状态、当前主力设备。
- 设备持有成本：日均、月均、扣除出售价格后的净成本。

图片策略：

- 继续沿用“标题解析 + 本地精确映射表 + 置信度回退”。
- 设备详情可以使用高置信度官方图。
- 匹配不到时使用本地占位图，不使用近似图。

### 8.5 Reports 页面

年度报告内容：

- 年度总现金支出。
- 年度硬件支出。
- 年度软件 / App 支出。
- 年度订阅支出。
- 年度 Store Credit 充值和消费。
- 年度新增设备。
- 年度最高单笔消费。
- 与上一年对比。

导出阶段：

1. 先支持 Markdown。
2. 再支持浏览器内 PNG 长图。
3. 最后评估 PDF。

### 8.6 Data Quality 页面

核心能力：

- 展示 issue 列表。
- 按严重程度和类型筛选。
- 展示涉及的交易、资产、订阅或流水。
- 支持 dismiss。
- 对可修正问题提供跳转或手动关联入口。

第一阶段只做诊断展示，第二阶段再做交互式修复。

### 8.7 Rules 设置页面

规则类型：

- 分类规则。
- 支付方式现金口径规则。
- Store Credit 口径规则。
- 排除规则。
- 订阅名称归一规则。
- 设备别名规则。

存储：

- `ruleSettings` 表。
- 默认规则在代码中。
- 用户规则覆盖默认规则。

## 9. 统计与选择器计划

建议新增选择器模块：


统计原则：

- 金额统一使用 decimal.js。

- 现金支出与账单项目价值继续分离。
- Store Credit 充值计入现金支出，Store Credit 消费不重复计入现金支出。
- 订阅累计支出以实际交易为准，Subscription History 只作为事件补充。

## 12. 风险与取舍

### 12.1 Apple CSV 字段变化

处理：

- 文件识别基于表头签名。
- parser 对字段缺失要降级。
- 缺关键字段时生成 data quality issue。

### 12.2 Store Credit 重复计入

处理：

- Store Credit 消费默认不计入现金支出。
- 充值才计入现金支出。
- 所有口径在 UI 中可查看说明。

### 12.3 订阅状态不准确

处理：

- Subscription History 只作为事件来源。
- 真实扣费以交易记录为准。
- 状态不确定时显示 unknown，不硬猜。

### 12.4 设备关联误判

处理：

- 序列号 / 订单号优先。
- 名称和日期只作为弱关联。
- 弱关联必须进入数据质量中心等待确认。

### 12.5 报告导出复杂度

处理：

- 先做 Markdown。
- PNG 长图放第二阶段。
- PDF 最后评估，不让导出阻塞核心页面。

6. V2-6 年度报告。
7. V2-7 规则配置。

原因：

- Store Credit 和订阅直接影响核心统计口径，优先级高。
- 数据质量中心应尽早出现，否则 V2 新数据源会产生无法解释的异常。
- 设备生命周期依赖更多关联规则，适合在数据质量框架之后扩展。
- 年度报告依赖前面所有统计模块，最后做更稳。

## 14. 第一轮建议实施包

如果确认进入 V2，建议第一轮只做一个闭环：

### 范围

- 文件签名识别重构。
- ZIP / 多 CSV 导入入口。
- Dexie version 2 新表。
- Store Credit Transaction History parser。
- Store Credit selectors。
- `/store-credit` 页面。
- 数据质量 issue 最小模型。

### 不包含

- 订阅中心。
- AppleCare。
- 年度报告导出。
- 规则配置页面。

### 验收

- 用户可以导入 ZIP 或多个 CSV。
- Store Credit 中心可以解释累计充值、已消费、估算余额。
- Dashboard 口径不被破坏。
- 未识别或无法关联的数据会出现在数据质量提示中。
