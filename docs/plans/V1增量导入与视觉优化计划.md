# V1 增量导入与视觉优化计划

## 1. 背景

当前 V1 已经完成基础链路：CSV 导入、App Store / Apple Store 解析、Dashboard、Ledger、Assets、IndexedDB 本地持久化和基础 Apple-like 视觉系统。

本次优化聚焦用户反馈的三个问题：

1. 导入只能替换全部数据，不支持硬件 CSV 和软件 CSV 分开增量导入。
2. Ledger 明细缺少硬件 / 软件图像，详情侧边栏也没有大图展示。
3. 当前卡片风格、圆角、动画和 Apple 官网质感仍不够。

本计划只覆盖下一轮优化，不直接进入 V2 的 ZIP、报告导出、订阅中心或云同步。

## 2. 优化目标

### 2.1 增量导入

允许用户多次导入不同目录下的 CSV：

- 第一次只导入 App Store 软件消费 CSV。
- 第二次再导入 Apple Store 硬件订单 CSV。
- 后续可以继续补充新的 CSV。
- 新导入数据与已有 IndexedDB 数据合并，而不是默认清空重建。
- 重复交易需要自动识别并跳过，避免重复统计。

### 2.2 Ledger 图像增强

在明细表和详情抽屉中加入交易图像：

- 硬件交易优先显示对应设备图片。
- 软件 / App / 订阅如果有图像，则显示对应图像。
- Ledger 表格第一列展示小图。
- 点击交易后，右侧详情抽屉展示大图。
- 没有匹配图像时，使用 Apple-like 的抽象占位图，而不是空白。

### 2.3 Apple-like 视觉升级

将当前基础卡片风格升级为更接近 Apple 官网的数据产品体验：

- 增加更明显但克制的圆角系统。
- 卡片需要有 Apple 官网式的空间感、柔和边界和轻微层次。
- 页面进入、卡片 hover、明细打开、图像出现需要有自然动画。
- Dashboard、Ledger、Assets 的视觉语言保持一致。
- 避免变成传统后台管理系统。

## 3. 设计原则

### 3.1 数据安全

所有新增能力仍保持本地优先：

- 不上传图片和 CSV 到服务端。
- 图片素材优先来自本地 `public/apple-assets/`。
- IndexedDB 保存交易、资产、导入批次和图片匹配关系。

### 3.2 统计正确性

增量导入必须优先保证不会重复计数：

- 同一 App Store 交易重复导入时不能产生重复交易。
- 同一 Apple Store 硬件订单重复导入时不能产生重复交易或重复资产。
- 增量导入后 Dashboard 统计必须基于合并后的唯一数据。

### 3.3 视觉一致性

新增图片、圆角、动画不能只在单个页面生效：

- Ledger 小图、详情大图、Assets 资产图需要复用同一套图像匹配逻辑。
- 卡片、按钮、标签、抽屉、表格行 hover 需要来自统一 tokens。
- 动画时长、缓动曲线和位移幅度需要集中定义。

## 4. 技术方案

## 4.1 增量导入方案

### 4.1.1 新增导入模式

在 `ImportPage.vue` 增加导入模式选择：

| 模式 | 行为 |
| --- | --- |
| 增量导入 | 默认模式。解析新 CSV 后与现有数据合并，跳过重复交易和资产。 |
| 替换全部 | 高风险操作。清空已有交易、资产和批次后重新导入。 |

默认选择「增量导入」，「替换全部」需要二次确认。

### 4.1.2 Store 层改造

当前 `transaction-store.ts` 已有：

- `replaceAll()`
- `clearAll()`
- `loadFromStorage()`

需要新增：

- `appendImport(transactions, assets, batch)`
- `mergeTransactions(existing, incoming)`
- `mergeAssets(existing, incoming)`

合并规则：

- 交易以 `AppleTransaction.id` 作为主唯一键。
- 资产以 `AppleAsset.id` 作为主唯一键。
- 新批次追加保存，不清空旧批次。
- 如果重复 ID 已存在，跳过 incoming 数据，并在导入摘要里记录重复数量。

### 4.1.3 IndexedDB 改造

当前 `storage/db.ts` 有 `replaceLedgerData()` 和 `clearLedgerData()`。

需要新增：

- `appendLedgerData(transactions, assets, batch)`

实现方式：

- 先查询现有 `transactions` 和 `assets` 的 ID 集合。
- 过滤重复数据。
- 使用 Dexie transaction 批量写入新增数据。
- 保存新的 `ImportBatch`。

### 4.1.4 ImportBatch 扩展

当前 `ImportedFileSummary` 只有：

- `rowCount`
- `parsedCount`
- `skippedCount`

建议扩展：

```ts
export interface ImportedFileSummary {
  path: string
  type: string
  rowCount: number
  parsedCount: number
  skippedCount: number
  duplicateCount?: number
  insertedCount?: number
}
```

用于让用户明确知道本次增量导入：

- 解析了多少行。
- 新增了多少条。
- 跳过了多少重复数据。

### 4.1.5 Import 页面交互

导入页需要改为「数据管理」体验：

- 顶部展示当前本地数据概况：
  - 总交易数
  - 资产数
  - 导入批次数
  - 最近导入时间
- 上传区域默认执行增量导入。
- 导入后展示「本次新增」和「重复跳过」。
- 清空数据继续保留，但作为低频危险操作。

## 4.2 图片能力方案

### 4.2.1 图片素材目录

新增公开静态资源目录：

```text
public/
  apple-assets/
    manifest.json
    hardware/
    software/
    placeholders/
```

`manifest.json` 用于描述图片匹配规则：

```json
[
  {
    "id": "iphone-default",
    "kind": "hardware",
    "category": "iPhone",
    "matchKeywords": ["IPHONE"],
    "thumbnail": "/apple-assets/hardware/iphone-default.png",
    "image": "/apple-assets/hardware/iphone-default.png",
    "source": "local-placeholder"
  }
]
```

V1 优化阶段先不抓取公网图片，不做版权风险不明确的自动下载。可以先用本地占位图和用户后续手动放入的图片。

### 4.2.2 新增图像模型

新增文件：

```text
src/features/media-assets/
  asset-image-model.ts
  image-manifest.ts
  image-resolver.ts
```

建议模型：

```ts
export interface AppleVisualAsset {
  id: string
  kind: 'hardware' | 'software' | 'placeholder'
  category?: string
  matchKeywords: string[]
  thumbnail: string
  image: string
  source?: string
}
```

### 4.2.3 图像匹配规则

统一由 `image-resolver.ts` 处理，避免页面各写一套逻辑。

匹配优先级：

1. 交易或资产手动指定的图片 ID。
2. `category + title + subtitle` 命中 manifest 关键词。
3. 硬件类别默认图：
   - iPhone
   - iPad
   - Mac
   - Apple Watch
   - AirPods
   - AppleCare
   - Accessory
4. 软件默认图：
   - App
   - Subscription
   - In-App Purchase
   - Store Credit
5. 通用占位图。

### 4.2.4 数据模型扩展

建议给交易和资产增加可选字段：

```ts
imageId?: string
imageUrl?: string
thumbnailUrl?: string
```

短期也可以不落库，直接运行时通过 resolver 计算。为了后续支持手动选择图片，建议保留 `imageId` 字段。

### 4.2.5 Ledger 页面改造

`LedgerPage.vue` 改造点：

- 表格最前面增加 64px 左右的图片列。
- 图片使用统一组件 `AppleAssetImage.vue`。
- 移动端卡片也展示缩略图。
- 点击行打开抽屉时，抽屉顶部展示大图。
- 大图下方再展示交易标题、金额、口径标签和原始字段。

新增组件建议：

```text
src/components/media/
  AppleAssetImage.vue
  AppleImageHero.vue
```

### 4.2.6 Assets 页面复用

`AssetsPage.vue` 当前使用抽象 CSS 设备轮廓。优化后：

- 优先展示 manifest 匹配到的设备图。
- 没有图时使用新的 Apple-like 占位图组件。
- 保持资产卡片的数据密度克制。

## 4.3 Apple-like 视觉升级方案

### 4.3.1 Tokens 调整

修改 `src/styles/tokens.css` 和 `tailwind.config.ts`。

建议新增：

```css
:root {
  --radius-card: 28px;
  --radius-panel: 22px;
  --radius-control: 999px;
  --ease-apple: cubic-bezier(0.2, 0, 0, 1);
  --duration-enter: 420ms;
  --duration-hover: 180ms;
  --shadow-card: 0 18px 48px rgba(0, 0, 0, 0.08);
  --shadow-card-hover: 0 28px 70px rgba(0, 0, 0, 0.12);
}
```

注意：

- 计划里之前有「卡片圆角 8px 或更少」的通用规范，但本项目明确追求 Apple 官网启发的产品陈列感，可以在页面级卡片使用更大的圆角。
- 表格行、按钮、输入框仍保持较小圆角，避免全局都变成胶囊。

### 4.3.2 AppleCard 升级

`AppleCard.vue` 需要支持更多视觉变体：

| 变体 | 用途 |
| --- | --- |
| `surface` | 普通页面区块 |
| `hero` | Dashboard 首屏 Hero Metric |
| `media` | 图片资产卡、Ledger 图片卡 |
| `quiet` | 数据质量提示、低优先级信息 |

交互：

- hover 时轻微上移 2px。
- shadow 轻微增强。
- 图片卡允许 `overflow-hidden`。
- 支持进入动画延迟。

### 4.3.3 动画系统

新增或增强：

```text
src/components/ui/AppleReveal.vue
src/styles/motion.css
```

动画类型：

- 页面进入：整体淡入 + 轻微上移。
- 卡片进入：stagger reveal。
- 表格行 hover：背景、图片缩放、金额强调。
- 抽屉打开：大图淡入 + scale 0.98 到 1。
- Dashboard 图表：保留 ECharts 自带动画，但颜色和节奏更克制。

动画约束：

- 不使用夸张弹跳。
- 不使用长时间视差。
- 不影响查账效率。
- `prefers-reduced-motion: reduce` 下关闭非必要动画。

### 4.3.4 页面级视觉调整

#### Dashboard

- Hero Metric 使用更接近 Apple 官网首页的信息层级。
- 首屏卡片圆角更大，背景更轻。
- 次级指标降低边框感，改为内部分区。
- 图表容器更少表格感。

#### Ledger

- 表格行变成更像 Apple 订单列表：
  - 图片在最左。
  - 商品名和所属服务有清晰层级。
  - 金额右对齐。
  - 标签弱化。
- 桌面端保留表格效率，移动端用卡片列表。
- 详情抽屉升级为「产品详情 + 原始字段」结构。

#### Assets

- 资产卡更像产品陈列：
  - 顶部大图。
  - 中部设备名称。
  - 底部购买信息和状态维护。
- 分类导航更接近 Apple 产品线切换。

## 6. 风险与取舍

### 6.1 增量导入风险

风险：如果不同 CSV 中同一交易的字段发生轻微变化，ID 可能不同，导致重复。

处理：

- V1 优先使用现有稳定 ID。
- 对 App Store 可追加 `Order Number + Date + Title + Amount` 的 fallback duplicate key。
- 对硬件订单继续使用当前硬件去重 key。
- 后续再做数据质量中心，让用户手动合并疑似重复项。

### 6.2 图片素材风险

风险：直接内置 Apple 官网产品图可能涉及公开分发授权问题。

处理：

- 本次先建立本地图片能力和 manifest。
- 默认使用自制占位图或用户本地放入的图片。
- 如果只用于个人本地项目，可以后续手动放入 Apple 官网素材，但不在代码里自动下载。

### 6.3 视觉升级风险

风险：过大的圆角和动画会降低账本工具的信息效率。

处理：

- Dashboard 和 Assets 可以更产品化。
- Ledger 保持查账效率，圆角和动画只增强感知，不牺牲密度。
- 表格、筛选控件和抽屉保持清晰、稳定、可扫描。

## 7. 需要确认的问题

实施前建议确认以下决策：

1. 目录名是否确认为 `docs/palns`，还是要改回 `docs/plans`。
2. 增量导入时，默认是否保留旧数据并自动跳过重复项。
3. 图片素材是否先使用本地占位图和 manifest，不自动抓取外部图片。
4. 是否允许在 `AppleTransaction` 和 `AppleAsset` 上增加 `imageId` 等可选字段，为后续手动指定图片做准备。

## 8. 推荐结论

建议下一步先实施「阶段 1：增量导入数据层」。

原因：

- 这是当前使用流程中最影响可用性的阻塞点。
- 它会改变数据写入方式，需要先稳定统计口径。
- 图片和视觉升级都建立在稳定的交易 / 资产数据之上。

增量导入通过后，再做图片 resolver 和视觉系统升级，可以减少返工。
