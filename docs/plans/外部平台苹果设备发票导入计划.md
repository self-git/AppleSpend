# 外部平台苹果设备发票导入计划

## 1. 背景

当前系统已经支持 Apple 官方数据导出的 App Store、Apple Store、订阅、余额、设备信息等 CSV / ZIP 导入。下一步需要支持用户在京东、天猫、淘宝等外部平台购买的 Apple 设备，使这些设备也能进入统一的消费账本和资产管理。

用户当前可提供的数据主要包括：

- 电子发票 PDF。
- 订单截图。
- 发票截图。

根据示例发票测试，京东 / 天猫类电子发票 PDF 多数可以直接抽取文本，不一定需要 OCR。发票字段中包含商品名称、规格型号、数量、金额、销售方、开票日期、订单号等，适合作为高准确率导入来源。

## 2. 目标

1. 支持导入外部平台购买的 Apple 设备。
2. 优先解析电子发票 PDF，截图 OCR 作为补充方案。
3. 将确认后的记录写入现有 `AppleTransaction` 和 `AppleAsset` 模型。
4. 保持本地优先，默认不上传发票、截图和识别文本。
5. 不保存 SN、IMEI 等敏感设备识别号原文。
6. 使用不可逆算法分别从 SN、IMEI 派生本地识别短码，用于同一设备的稳定关联。
7. 所有识别结果先进入预览确认，不直接静默入库。

## 3. 非目标

- 不做京东、淘宝、天猫自动登录。
- 不做网页爬虫和订单接口抓取。
- 不做发票真伪联网验真。
- 不保存完整 SN、IMEI、身份证号、手机号等敏感字段。
- 不承诺自动识别所有图片截图，首版只保证电子发票 PDF 的稳定导入。

## 4. 数据来源优先级

### 4.1 电子发票 PDF

优先级最高。

可抽取字段：

- 发票号码。
- 发票代码，旧版发票可能存在。
- 开票日期。
- 购买方名称。
- 销售方名称。
- 销售方纳税人识别号。
- 项目名称 / 商品名称。
- 规格型号。
- 单位。
- 数量。
- 单价。
- 金额。
- 税率。
- 税额。
- 价税合计。
- 备注中的订单号、SN、IMEI、补贴金额、用户实付金额、订单金额。

### 4.2 发票图片 / 订单截图

作为补充来源。

可抽取字段：

- 平台。
- 店铺名。
- 订单号。
- 下单时间。
- 商品名。
- 商品规格。
- 实付款。
- 优惠 / 补贴信息。
- 支付方式。

截图结构不稳定，首版应进入“低置信度草稿”，必须人工确认。

## 5. 字段映射

### 5.1 交易模型

确认后生成 `AppleTransaction`：

```ts
{
  source: 'external_retail',
  date: invoiceDate || orderDate,
  title: normalizedProductName,
  subtitle: platform + sellerName,
  category: classifyHardware(normalizedProductName),
  amount: cashAmount,
  currency: 'CNY',
  paymentMethod: platform,
  orderNumber,
  rawType: 'External Retail Invoice',
  cashImpact: true,
  billValueImpact: true,
  isFree: false,
  isRefund: amount < 0,
}
```

### 5.2 资产模型

确认后生成 `AppleAsset`：

```ts
{
  name: normalizedProductName,
  category: classifyHardware(normalizedProductName),
  purchaseDate: invoiceDate || orderDate,
  purchasePrice: cashAmount,
  currency: 'CNY',
  orderNumber,
  sourceTransactionId,
  status: 'unknown',
  note: platform + sellerName,
}
```

### 5.3 金额口径

金额需要区分两层：

- `cashAmount`：用户真实现金支出，优先取“用户实付金额”，其次取“价税合计”，再次取“订单金额”。
- `billValueAmount`：设备票面价值，优先取“价税合计 / 订单金额”。

示例：

- 如果发票备注包含“用户实付金额 7499.00，订单金额 7999.00，政府补贴 500.00”，现金支出应为 7499.00，设备票面价值可保留为 7999.00。
- 如果发票只有“价税合计 ¥8099.00”，现金支出和票面价值都暂取 8099.00。

## 6. SN / IMEI 隐私方案

### 6.1 原则

- 不保存原始 SN。
- 不保存原始 IMEI。
- 不保存包含原始 SN / IMEI 的完整 OCR 文本。
- 不在 UI 默认展示 SN / IMEI。
- 不把 SN / IMEI 发送给外部服务。

### 6.2 本地不可逆识别短码

严格来说，这里不应使用“可解密加密”，而应使用不可逆 HMAC。原因是：如果保存可解密密文，密钥泄露后仍可还原 SN / IMEI；而 HMAC 只能验证同一输入是否一致，不能反推出原始识别号。

建议算法：

```text
deviceSecret = 首次运行时在本地生成的 256-bit 随机密钥
normalizedSn = 去空格、转大写、去分隔符后的 SN
snDigest = HMAC-SHA-256(deviceSecret, "sn:" + normalizedSn)
snFingerprint = "SN-" + Base32(snDigest).slice(0, 12)

normalizedImei = 去空格、只保留数字后的 IMEI
imeiDigest = HMAC-SHA-256(deviceSecret, "imei:" + normalizedImei)
imeiFingerprint = "IMEI-" + Base32(imeiDigest).slice(0, 12)
```

示例输出：

```text
SN-7K3P9Q2M4X8A
IMEI-2R8M6Q4Y9N3C
```

### 6.3 存储规则

只保存：

- `deviceFingerprint`：设备级主短码，优先使用 `snFingerprint`，没有 SN 时可使用第一个 `imeiFingerprint`。
- `snFingerprint`：SN 派生的不可逆短码。
- `imeiFingerprints`：IMEI 派生的不可逆短码数组，支持双卡设备的 IMEI1 / IMEI2。
- `fingerprintAlgorithm`：例如 `hmac-sha256-v1`。
- `fingerprintSources`：例如 `['invoice_sn', 'invoice_imei1', 'invoice_imei2']`。

不保存：

- `serialNumber` 原文。
- `imei1` 原文。
- `imei2` 原文。
- 未脱敏备注全文。
- 未脱敏 OCR 全文。

### 6.4 无 SN / IMEI 时的回退

如果发票或截图没有 SN / IMEI：

- 不生成 `deviceFingerprint`。
- 仍可通过 `platform + orderNumber + invoiceNumber + productName + amount + date` 做订单级去重。
- UI 提示“缺少设备级识别码，只能按订单维度关联”。

### 6.5 IMEI 处理

IMEI 比 SN 更敏感，但用户已确认 IMEI 需要参与本地识别。首版处理方式为：只保存 IMEI 派生的 HMAC 短码，不保存 IMEI 原文。

处理方式：

- 解析阶段可短暂识别 IMEI，用于生成 `imeiFingerprints` 和脱敏文本。
- 入库前必须丢弃 IMEI 原文。
- 支持 IMEI1 / IMEI2 分别生成短码。
- UI 默认只展示短码，不展示原始 IMEI。

## 7. 文本脱敏规则

导入草稿和入库记录都必须先脱敏。

建议脱敏字段：

- `SN:<value>` -> `SN:[REDACTED]`
- `IMEI1:<value>` -> `IMEI1:[REDACTED]`
- `IMEI2:<value>` -> `IMEI2:[REDACTED]`
- 身份证号 -> `[ID_REDACTED]`
- 手机号 -> `[PHONE_REDACTED]`

注意：

- 原始 PDF 文件不复制到 IndexedDB。
- 如果需要保存文件引用，只保存本地导入批次的文件名，不保存绝对路径。
- 如果需要保存抽取文本，只保存脱敏后的 `redactedText`。

## 8. 解析流程

### 8.1 PDF 文本型发票

1. 用户上传 PDF。
2. 使用 PDF 文本抽取能力读取正文。
3. 判断是否为发票：
   - 包含“电子发票”或“增值税电子普通发票”。
   - 包含“价税合计”。
   - 包含“项目名称”或“货物或应税劳务、服务名称”。
4. 抽取基础字段。
5. 抽取项目明细。
6. 从备注中抽取订单号、用户实付金额、订单金额、补贴金额、SN、IMEI。
7. 使用 SN / IMEI 生成本地 HMAC 短码，立即丢弃原始 SN / IMEI。
8. 对文本做脱敏。
9. 生成导入草稿。
10. 用户确认后写入交易和资产。

### 8.2 图片 / 扫描型发票

1. 用户上传图片或扫描型 PDF。
2. 本地 OCR。
3. 使用同一套字段抽取规则。
4. 标记 `confidence`。
5. 低置信度字段要求用户手动确认。

### 8.3 订单截图

1. OCR 提取订单号、商品名、实付款、店铺名、下单时间。
2. 不生成发票字段。
3. 标记为“订单证据”。
4. 如果和发票订单号匹配，则合并到同一草稿。

## 9. 导入预览页面

新增导入预览步骤，字段分区展示：

- 商品信息：设备名、类别、规格、数量。
- 金额信息：用户实付、订单金额、价税合计、补贴。
- 渠道信息：平台、销售方、订单号、发票号码。
- 日期信息：开票日期、下单日期。
- 隐私信息：只显示 `deviceFingerprint`、`snFingerprint`、`imeiFingerprints`，不显示原始 SN / IMEI。
- 置信度：自动识别字段显示高 / 中 / 低置信度。

用户可执行：

- 修改设备名。
- 修改类别。
- 修改金额口径。
- 修改日期。
- 合并重复草稿。
- 忽略某张发票。
- 确认导入。

## 10. 去重策略

### 10.1 设备级去重

如果存在 `deviceFingerprint`：

```text
deviceFingerprint
```

命中即提示可能是同一设备。

### 10.2 订单级去重

没有 `deviceFingerprint` 时：

```text
platform + orderNumber + invoiceNumber + normalizedProductName + amount + date
```

### 10.3 弱去重

当订单号或发票号缺失：

```text
sellerName + normalizedProductName + amount + date
```

弱去重只提示，不自动跳过。

## 15. 风险与处理

- 发票格式差异：使用多规则解析，无法识别时进入手动确认。
- 优惠 / 补贴金额口径复杂：预览页明确区分现金支出和票面价值。
- SN / IMEI 泄露风险：原文不落库，入库前强制脱敏。
- 同一设备多张凭证：使用 `deviceFingerprint`、`snFingerprint`、`imeiFingerprints` 和订单号做重复提示。
- 浏览器端 PDF 文本抽取兼容性：优先选择成熟库，无法抽取时回退 OCR。

## 16. 确认点

实施前需要确认：

1. 已确认接受 `external_retail` 作为统一来源，平台放在 `platform` 字段中。
2. 已确认现金支出优先取“用户实付金额”。
3. 已确认 SN 使用 HMAC-SHA-256 生成不可逆短码，而不是可解密加密。
4. 已确认 IMEI 和 SN 都参与识别信息存储，但只保存 HMAC-SHA-256 派生的不可逆短码，不保存原文。
5. 已确认第一轮只实现电子发票 PDF，不做截图 OCR。
